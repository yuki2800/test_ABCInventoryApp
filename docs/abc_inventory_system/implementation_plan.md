# ABC在庫管理システム 実装計画

## 目的
アーキテクチャ（クリーンアーキテクチャ、ドメイン駆動設計）の学習を目的とした保守性の高い在庫管理システムの構築。

## 概要と技術スタック
業務システムの主流構成を取り入れつつ、学習に適したモダンな技術スタックを採用します。

*   **フロントエンド**: Next.js (App Router), React, TypeScript
*   **バックエンド**: Node.js, TypeScript (Next.jsのAPI Routesを使用し、バックエンドのロジックとしてクリーンアーキテクチャを適用)
*   **データベース (ストレージ)**: **インメモリ (In-Memory)**。データベースは使用せず、バックエンドのメモリ内 (変数/Map等) でデータを保持します。再起動でデータは初期化されます。
*   **スタイリング**: Tailwind CSS (Next.jsデフォルト) / Vanilla CSS (モダンで美しい独自のUIシステム)

## 設計思想 (アーキテクチャ)

### クリーンアーキテクチャ
バックエンド（API）のロジックを以下の4層に分割します。
1.  **Domain Layer (エンティティ層)**: 中心となる業務モデル。外部に依存しない。
2.  **Use Case Layer (ユースケース層)**: アプリケーションのビジネスルールフローを制御。
3.  **Interface Adapter Layer (インターフェースアダプタ層)**: APIコントローラーやデータベースへのリポジトリ実装（「インメモリストレージ」へのアクセス実装）。
4.  **Framework & Driver Layer (フレームワーク層)**: Next.jsのルーティングなど。

### ドメイン駆動設計 (DDD) の強みを示すポイント
「製品マスタごと（製品カテゴリごと）に処理の分岐を作成する」という要件に対して、ドメイン層のエンティティにポリモーフィズム (多態性) を持たせます。
*   **カテゴリの例**: `STANDARD` (通常品), `PERISHABLE` (生鮮食品), `HAZARDOUS` (危険物)
*   **ドメインロジック**:
    *   `PERISHABLE`: 受入時に「消費期限」の入力が必須であり、現在日付より過去であればエラーにする。
    *   `HAZARDOUS`: 払出時に特別な承認フラグや安全チェックの基準を満たす必要がある。
    *   これらをユースケース層で `if/else` で分岐するのではなく、Domain Entity 自身に検証ロジック (`validateReceipt`, `validateIssue`) を持たせることで、DDDの強み（凝集性の高さとカプセル化）を明確に表現します。

### Reactの強みを示すポイント
*   **コンポーネントの再利用性**: 検索バー、データテーブル、各種フォームを汎用的なコンポーネントとして分割。
*   **状態管理とレンダリング**: 検索キーワードに応じた在庫一覧のリアルタイムフィルタリングや、Optimistic UI的なシームレスな更新。
*   **Custom Hooks**: バックエンドAPIとの通信をカプセル化し、UIコンポーネントからロジックを切り離す。
